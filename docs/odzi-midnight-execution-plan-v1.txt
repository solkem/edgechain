Odzi x Midnight Execution Plan (Lean Startup, Field-First)

Owner: EdgeChain team
Location: Odzi, Zimbabwe
Plan start date: TBD (set with cooperative operations calendar)
Plan horizon: 6 weeks from kickoff

==================================================
0) Objective
==================================================

Validate a real-world Midnight use case with farmers before seeking scale funding.

What success means in this phase:
- farmers use a field recording tool to capture real harvest-to-auction events;
- app returns deterministic receipts for each record (Phase A off-chain);
- trust + repeat behavior are measured and improving;
- the pilot produces clean evidence for Phase B Midnight anchoring.
- field tool performs reliably on low-cost Android phones.
- daily Merkle roots are anchored via an initial custodial desktop operator flow with verifiable tx logs.

Out of scope in this phase:
- lending, credit underwriting, or DeFi product rollout;
- complex token mechanics;
- per-farmer IoT hardware deployment.

==================================================
0.1) Artifact Boundaries (Do Not Conflate)
==================================================

A) Field recording tool (pilot-critical, must work at kickoff)
- Start session -> record event -> receipt -> history
- Android-runnable, offline-tolerant, deterministic receipt generation
- optimized for low-cost Android constraints (data, memory, battery)

B) Learning Explorer (reviewer/funder education artifact)
- Not required for kickoff-day farmer operations
- Should be updated post-pilot with real data

C) Midnight integration layer
- Phase A: off-chain deterministic receipt generation + custodial desktop anchoring lane
- Phase B: daily root anchoring + inclusion proofs under a mobile-constrained trust model

==================================================
0.2) Mobile-Constrained Trust Model (MCTM)
==================================================

Given current device/wallet constraints:
- farmer phones are not required to run native trustless wallet flows
- farmers use mobile app for event capture + receipts
- operator desktop performs daily Midnight root anchoring in an initial custodial model
- each receipt maps to an inclusion proof under anchored root

Phase A trust posture:
- custody exists at anchor submission, but verification outputs remain trust-minimized
- deterministic receipts, tx-hash references, and inclusion proofs remain independently checkable
- two-person anchor control and anchoring SLA are mandatory controls

Reference:
- docs/MOBILE_CONSTRAINED_TRUST_MODEL.md

1) Phase A Field Recording Tool Scope (v0)
==================================================

Farmer-facing workflow:
1. Start session
   - Farmer ID (or coop ID mapping), consent checkbox.
2. Record event
   - Event type: HARVEST, BALE_PREP, DISPATCH.
   - Required fields:
     - crop lot ID
     - date/time
     - notes (short)
   - Conditional fields:
     - BALE_PREP: bale tag, est weight, est grade
     - DISPATCH: destination floor, transporter, departure time
3. Receipt screen
   - Record ID
   - Timestamp
   - Receipt fingerprint (deterministic hash-like ID)
   - "Show this receipt when you need to verify this record."
4. History view
   - Last 10 records with receipt IDs and status.

Technical note:
- Phase A receipts are off-chain deterministic artifacts (not on-chain proofs yet).
- Phase B anchors these artifacts to Midnight commitments and nullifier rules.
- In MCTM, anchoring runs through a controlled desktop custodial lane until mobile-native wallet flows are practical.

==================================================
4) Week-by-Week Lean Plan
==================================================

Week 0 (Kickoff Week): Problem + Workflow Fit
- Goal: prove farmers can use and understand receipt workflow.
- Outputs:
  - baseline data collected
  - first trust/behavior signal
  - top 5 friction points list

Week 1: Activation Reliability
- Goal: raise valid submissions and reduce support burden.
- Actions:
  - simplify form labels
  - improve error messages
  - retrain on most confusing step

Week 2: Trust Reinforcement
- Goal: ensure farmers perceive receipts as credible proof.
- Actions:
  - add receipt lookup demonstration
  - run "verify my record" exercises in group
  - start daily root anchoring runbook and anchor log publication
  - enforce two-person custodial anchor sign-off process

Week 3: Behavior Stability
- Goal: consistent weekly usage without heavy hand-holding.
- Actions:
  - lightweight reminders via liaison
  - measure drop-off reasons by segment

Week 4: Anchoring Integrity + Optional Wallet Lane
- Goal: validate anchoring reliability and optional connector lane where feasible.
- Actions:
  - inclusion proof spot-checks against anchored roots
  - publish reproducibility check outcomes for sampled receipts
  - optional wallet/connector trials on extension-capable devices only

Week 5: Evidence Packaging
- Goal: produce readiness dossier for Midnight discussion.
- Outputs:
  - KPI trend summary
  - user evidence and quotes (anonymized)
  - product changes made and rationale
  - clear funded next-step proposal

==================================================
5) Metrics and Gates
==================================================

Core gates:
- Activation (week 1): >= 75%
- Retention (week 4): >= 40% (recalibrate after baseline)
- Retention (week 6): >= 35% (recalibrate after baseline)
- Completeness: >= 90%
- Invalid rate: <= 5%
- Avg trust score: baseline + >= 0.7 lift by week 6
- Trust score 4 or 5: >= 60% (recalibrate after baseline)
- Anchoring coverage: >= 90% of active pilot days
- Anchoring timeliness: >= 80% anchored within 24h
- Inclusion proof verification success: >= 99% on sampled checks

Optional lane metrics (not blocking GO in this phase):
- Wallet/connector setup success on compatible devices
- Sign-without-help in small champion subgroup

Custodial control metrics (blocking in this phase):
- Anchor sign-off compliance: >= 95%
- Receipt reproducibility on sampled checks: >= 99%

Calibration rule:
- Treat week-0 baseline as denominator context.
- Final thresholds are locked after baseline review and documented in tracker notes.

Decision logic:
- GO: all critical trust/behavior/data gates pass.
- ITERATE: one critical gate missed, no critical incident.
- NO-GO: trust failure or safety/compliance critical issue.

8) Risk Controls
==================================================

Hard stops:
- any report of lost funds
- unresolved data tampering allegation
- coercion or misleading finance claims

Mitigations:
- explicit non-financial consent language
- backup paper capture flow
- immediate incident log and escalation owner
- custodial anchor controls: dual sign-off + tx hash publication + daily exception review
